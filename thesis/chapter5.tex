\chapter{Educational Curriculum Design} \label{ch5_curriculum}
\markboth{Educational Curriculum Design}{Educational Curriculum Design}


In this chapter, having established the technical implementation of the modular blockchain system in Chapter \ref{ch4_implementation}, this chapter presents a comprehensive \textbf{5-Module Laboratory Curriculum}. The core contribution of this thesis is not only the blockchain software developed, but the educational curriculum that utilizes this software to bridge the gap between the theoretical distributed systems concepts and the practical engineering skills in distributed systems.


The curriculum is designed to transform the learning process from understanding the theoretical aspects to applying what they learned in their courses or projects, actively engaging with the underlying mechanics of consensus, network propagation, and state management. By interacting with the blockchain system developed in the previous chapter, students can transition from passive knowledge consumers to active system architects.


\section{Approach and Methodology} \label{ch5_pedagogy}

The design of the laboratory exercises is inspired by the theory of Constructionist Learning, proposed by Seymour Papert. This theory states that learning is most effective when students are actively engaged in constructing tangible objects in the real world, rather than acquiring knowledge through passive transmission or consumption \cite{papert1991situating}. In the context of Computer Science education, this necessitates moving beyond theory, static simulations, or black-box interactions with production networks (such as sending transactions on a public Testnet) toward deep system undrstanding allowing students to be able to fully understand the concepts by practice and gain code-level modifications experience.

To facilitate this, the curriculum adopts a Problem-Based Learning methodology. Rather than following specific instructions and editing a finished system, students are presented with engineering challenges that mirror real-world distributed systems problems. To ensure a scientific approach, each of the five modules follows a standardized structure inspired by the Active Learning framework \cite{freeman2014active}:


\begin{enumerate}
    \item \textbf{Problem Statement:} Introduces a specific theoretical challenge (e.g., "How do decentralized nodes agree on time?") or an operational deficiency (e.g., "How does the network behave when a peer disconnects?").
    \item \textbf{Learning Objectives:} Defines specific technical competencies the student will acquire by completing the module, aligned with the implementation details discussed in Chapter \ref{ch4_implementation}.
    \item \textbf{Laboratory Tasks:} This is the core phase, where students are tasked to find a solution and start modifying in the codebase, consensus parameters, altering the network module, or implementing new transaction types to resolve the stated problem.
    \item \textbf{Analysis of Findings:} states an observation of the system's behaviors in relation to the actions taken from the previous steps, such as throughput, latency, or change in the state of the system.
    \item \textbf{Learning Outcome:} In this phase, students correlate and connect their experimental results with the theoretical bounds established in Chapter \ref{ch2_theoretical_framework}, solidifying their gained knowledge of distributed systems and blockchains.
\end{enumerate}

This structured approach addresses the educational disconnect identified in Chapter \ref{ch1_introduction}, providing a safe environment in which breaking and changing the system is facilitated through a constructive learning mechanism, rather than a failure in the system, thus leading to a further engagement of both students and teachers in the learning process.

\section{Repository Structure and Setup} \label{ch5_repo_structure}

To support the modular nature of the curriculum, a dedicated GitHub repository has been organized to provide distinct entry points for each learning module. \url{https://github.com/Mostamhd/blockchain-educational-system} \label{ch5_repo_url}

The repository is structured to provide the necessary code base to start working on each module for the students. For each module, two directories are provided:
\begin{itemize}
    \item \textbf{Starter/}: This directory contains the baseline code required to initiate the module. Although the provided code is functional, it is intentionally incomplete. It serves as a starting point from which students are expected to implement the required features and address the problem statement associated with the module.

    \item \textbf{Solution/}: This directory contains a reference implementation corresponding to each module. It is intended to support self-assessment and solution validation. Due to space limitations, the complete solution implementations are not reproduced in full within this thesis. Instead, only the most critical modifications and design decisions are discussed, with the full reference implementations available in the accompanying repository.
\end{itemize}

\textbf{Note:} While the full solutions are provided to ensure no student remains blocked, they are strongly encouraged to attempt each task independently before consulting the \texttt{Solution/} directory. The primary educational value of the curriculum lies in the process of system design, implementation, and debugging, rather than in the examination of completed code alone.

\section{Module 1: Network Fundamentals} \label{ch5_module1}

This module focuses on two foundational building blocks of blockchain technology: creating a custom blockchain via forking and understanding Peer-to-Peer (P2P) discovery. Students are provided with the \textit{Starter Code} for the node software, which contains the core blockchain logic but lacks the specific configuration and implementation required to solve the tasks. The student's role is to act as a system architect, establishing a fork of the blockchain network and to understand how the network's peers discovery layer works to be able to create a new node and link it to the blockchain using the network's independent nodes.

\subsection{Problem Statement}
To facilitate a comprehensive understanding of blockchain networks, two fundamental challenges must be addressed:

\begin{enumerate}
    \item \textbf{Blockchain Forking:} How can a distinct blockchain network be instantiated such that it is cryptographically incompatible with other networks operating the same software version?
    
    \item \textbf{Peer Discovery Mechanism:} How do initially isolated nodes discover other participants within the network and establish peer-to-peer communication channels?
\end{enumerate}

\subsection{Learning Objectives}
By completing this module, students will be able to:
\begin{itemize}
    \item Execute a \textbf{Hard Fork} by modifying the Genesis configuration, thereby ensuring the cryptographic isolation of their private network.
    \item Understand the mechanics of \textbf{Peer-to-Peer Discovery} and how nodes bootstrap connections to an existing blockchain.
    \item Utilize \textbf{Docker} and \textbf{Docker Compose} to manage containerized nodes, gaining practical experience with containerization and node discovery in P2P systems.
\end{itemize}

\newpage

\subsection{Tasks}

\textbf{Task 1: The Hard Fork (Genesis Configuration)} \\
To establish a unique blockchain network identity, perform the following steps:
\begin{enumerate}
    \item Open the file \texttt{blockchain/block/block.py}.
    \item Locate the method \texttt{genesis()}.
    \item Modify the second parameter of the \texttt{Block} constructor (the \texttt{last\_hash} field) from the default value to a unique identifier (e.g., \texttt{"genesis\_group\_05"} or your Student ID).
    \item Save the file. This change ensures that the hash of Block 0 is mathematically distinct from the original codebase, effectively creating a hard fork.
\end{enumerate}

\textbf{Task 2: Network Node Addition} \\
Extend the existing network nodes by adding a new node, \texttt{node\_new}. You may choose to run the node manually via the terminal or orchestrate it via Docker.
\begin{enumerate}
    \item \textbf{Configuration:} Open the \texttt{docker-compose.yml} file.
    \item \textbf{Service Definition:} Define a new service entry named \texttt{node\_new}.
    \item \textbf{Port Mapping:} Map a unique host port to the container's internal P2P port.
    \item \textbf{Discovery:} Update the environment variables to ensure the new node is discoverable:
    \item \textbf{Execution:} Run \texttt{docker-compose up} and verify in the logs that \texttt{node\_new} successfully establishes a handshake with the existing peers.
\end{enumerate}

\subsection{Reference Solution \& Analysis}
The complete reference implementation for this module is archived in the accompanying repository, specifically within the \texttt{Module 1} directory (refer to Section \ref{ch5_repo_structure} for repository structure).\newline

\textbf{1. The Hard Fork:}
By changing the second parameter in the Genesis block to "genesis\_new\_student" this change in terms serves as the anchor for the entire blockchain. In the validation logic (\texttt{blockchain.py}), nodes verify that a new block links cryptographically to the previous one. By changing the genesis data the hash of the block, and subsequently all the hashes of the following blocks. Consequently, any node running the original code will calculate a different hash chain, which in turn successfully creates a new isolated blockchain network with a completely different chain of blocks.

\begin{lstlisting}[language=Python, caption=Genesis Block Modification, label=lst:mod1_genesis]
@staticmethod
def genesis():
    return Block([], "genesis_new_student", "genesis", 0)
\end{lstlisting}

\textbf{2. Network Node Addition:}
Two approaches may be used to add a node to the system. The first approach consists of executing the \texttt{run\_node.py} script with the appropriate parameters. However, this method requires that all nodes on the same machine to be executed in separate terminal sessions. Although functional, this approach can be more difficult to manage as the number of nodes grows.

The second approach, which is adopted in this work, relies on a \texttt{docker-compose} configuration to orchestrate multiple isolated nodes. Each node is represented as a Docker service. Implementing this approach requires the definition of specific environment variables to allow the peer-to-peer (P2P) module to correctly resolve and map Docker container IP addresses and ports to the corresponding network interfaces on the machine. The following configuration example demonstrates the correct method for creating a node in the system using docker-compose.

\begin{lstlisting}[language=yaml, caption=Docker Compose Solution, label=lst:mod1_docker]
node_new:
    build: ./blockchain
    command: python run_node.py --ip=0.0.0.0 --node_port=8015 
             --api_port=8055 --key_file=./keys/node_new_key.txt
    entrypoint: ['/usr/src/node/entrypoint.sh']
    volumes:
      - ./blockchain:/usr/src/node
    environment:
      - USE_DOCKER=True
      - DOCKER_NODE_CONTAINER_NAMES=node_one,node_two,node_new
      - DOCKER_NODE_PORTS=8010,8011,8015
    labels:
      com.datadoghq.ad.logs: '[{"service": "blockchain_node", 
                                "source": "python"}]'
    ports:
      - 8015:8015
      - 8055:8055
\end{lstlisting}

\textbf{Technical Justification of Configuration Parameters:}
\begin{itemize}
    \item \texttt{command}: Defines the runtime arguments. \texttt{--ip=0.0.0.0} binds the node to all network interfaces within the container, which is strictly necessary for it to receive external TCP connections from other Docker containers.
    \item \texttt{environment}: The environment variables are critical for the \textbf{Peer Discovery} mechanism. In \texttt{p2p\_communication.py}, the \texttt{SocketConnector} class parses these lists to map container names (e.g., `node\_one`) to their respective ports. This allows the new node to identify other nodes in the network and initiate the first handshake.
    \item \texttt{ports}: Maps the internal container ports to the host machine.
\end{itemize}

\subsection{Analysis of Findings}
Upon the completion of the task, students perform the following verifications:

\begin{itemize}
    \item \textbf{Fork verification:} Students compare the "Block Height" and "Last Hash" in the API of their new node versus a node from the original unmodified codebase. They observe that while both nodes are running both nodes run on different blockchains. This confirms that the modification to the Genesis configuration isolated the network, preventing the "Foreign" chain from corrupting the local ledger.
    \item \textbf{Connectivity Verification:} Students execute \texttt{docker-compose up} and inspect the logs. A successful node creation is indicated when \texttt{node\_new} logs a "Handshake" message, confirming it has established a connection with the other nodes in the blockchain network.
\end{itemize}

\subsection{Learning Outcome}
This module attempts to help students with demonstrating the importance of hashing in blockchains. Students are guided to learn that a blockchain's identity is not defined by the software running it, but rather by the shared history (Blocks) agreed upon by the network peers. Furthermore, the exercise provides practical experience in Infrastructure setup and network topology, by showing how creating a node requires an understanding of the underlying P2P networking layer and node discovery mechanisms in the distributed blockchain systems.

\section{Module 2: Consensus Mechanisms} \label{ch5_module2}

Having established a working distributed network in Module 1, the focus shifts to the mechanism of consensus in decentralized systems. As explained in \ref{ch2_consensus} The \textit{Starter Code} for this module includes the fully containerized setup from \ref{ch4_implementation}, which operates on a simplified POS consensus model. Students are tasked to redesign the system's consensus algorithm to help with achieving consensus in distributed systems, secure the network against attacks, and solve the Byzantine fault tolerance problem. While students are encouraged to explore various algorithms (e.g., Delegated Proof of Stake, Proof of Elapsed Time), this guide focuses on implementing Proof of Work (PoW) as it's still the most used and battle-tested consensus algorithm to date, and due to its educational value in demonstrating the trade-off between computational intensity and scalability.

\subsection{Problem Statement}
In an open, permissionless network where nodes can join anonymously, two critical vulnerabilities must be addressed:
\begin{enumerate}
    \item \textbf{The Sybil Attack:} Without a cost to participate, a single malicious actor can spawn thousands of nodes to gain a majority influence over the network, as mentioned in section \ref{ch2_security_risks}.
    \item \textbf{Leader Selection:} How can the network select a unique leader to propose the next block in a deterministic way without a central coordinator?
\end{enumerate}

\subsection{Learning Objectives}
By completing this module, students will be able to:
\begin{itemize}
    \item Analyze the architecture of the \texttt{Blockchain} class and design a method to switch the consensus algorithms in the blockchain prototype system built in chapter \ref{ch4_implementation} to the POW consensus algorithm explained in section \ref{ch2_consensus_pow}.
    \item Design and implement a cryptographic \textbf{Proof of Work} algorithm, translating theoretical concepts (Nonce, Difficulty, Target) into executable Python code.
    \item measure and analyze the impact of consensus choices on system performance.
\end{itemize}

\newpage

\subsection{Tasks}

\textbf{Task 1: The Consensus Implementation} \\
To secure the network, you must change the consensus algorithm and implement a Proof of Work algorithm to achieve consensus in the blockchain network.
\begin{enumerate}
    \item Create a new class \texttt{ProofOfWork} (e.g., in \texttt{blockchain/consensus/proof\_of\_work.py}) that adheres to the system's consensus interface.
    \item Implement the \texttt{mine(block)} method to perform the "Mining" process described in Section \ref{ch2_consensus_pow}. This method must:
    \begin{itemize}
        \item Initialize the block's \texttt{nonce} to 0.
        \item Execute a \texttt{while} loop that calculates the cryptographic hash of the block.
        \item Check if the resulting hash meets the difficulty target (e.g., starts with four leading zeros: \texttt{'0000'}).
        \item Increment the \texttt{nonce} and repeat until a valid hash is found.
    \end{itemize}
\end{enumerate}

\textbf{Task 2: The Mining Loop Integration} \\
The default node behavior is designed for Proof of Stake (Time-Bound), which waits for specific slots. You must alter the \texttt{node.py} logic to support the continuous, CPU-Bound nature of Proof of Work.
\begin{enumerate}
    \item Open the \texttt{blockchain/node/node.py} file and locate the \texttt{produce\_block} method.
    \item Modify the production loop to check which consensus strategy is currently active.
    \item Implement the conditional logic for PoW:
    \begin{itemize}
        \item Instead of waiting for a turn, the node must continuously call the \texttt{mine()} method created in Task 1.
        \item Ensure the system blocks (waits) until the mining function returns a valid block.
        \item Immediately broadcast the mined block to the network upon success.
    \end{itemize}
\end{enumerate}


\subsection{Reference Solution \& Analysis}
The complete reference implementation for this module is archived in the accompanying repository, specifically within the \texttt{Module 2} directory (refer to Section \ref{ch5_repo_structure} for repository structure).\newline

\textbf{1. Proof of Work Implementation:}
The solution involves creating a class that wraps the hashing logic in a while loop.

\begin{lstlisting}[language=Python, caption=Proof of Work Mining Logic, label=lst:mod2_pow_class]
class ProofOfWork:
    def __init__(self, difficulty=4):
        self.difficulty = difficulty

    def mine(self, block):
        block.nonce = 0
        while True:
            hash = block.hash_block()
            if hash.startswith('0' * self.difficulty):
                return block
            block.nonce += 1
\end{lstlisting}

The \texttt{mine} method models the block creation process as a probabilistic brute-force search problem. The \emph{difficulty} parameter serves as a regulatory control variable: incrementing the difficulty by one results in an exponential increase in the expected number of iterations required to discover a valid solution.

As a consequence, the effective \enquote{cost} of executing the attacks described in Section~\ref{ch2_security_risks} becomes proportional to the attacker’s available physical computing resources and the associated hardware costs, rather than merely the number of IP addresses under the attacker’s control.

\textbf{2. Node Integration:}
The Node class must be updated to handle the blocking nature of the mining process.

\begin{lstlisting}[language=Python, caption=Integrating the Mining Loop, label=lst:mod2_node_integration]
def run_produce_block(self):
    while self.keep_running:
        if isinstance(self.blockchain.consensus, ProofOfWork):
             # Create candidate block
            block = self.blockchain.create_block(
                self.transaction_pool.transactions, self.wallet
            )
            mined_block = self.blockchain.consensus.mine(block)
            
            if mined_block:
                self.p2p.broadcast(mined_block)
        else:
            # Default behavior (Sleep-based)
            time.sleep(10)
\end{lstlisting}

\subsection{Analysis of Findings}
Students validate their implementation by running the network and observing the container metrics.

\begin{itemize}
    \item \textbf{Resource Consumption:} Unlike Module 1, where CPU usage was near minmal, the PoW implementation causes an immediate CPU spike. This in terms proves that the security of the network relies on the physical energy process.
    \item \textbf{Difficulty Correlation:} By adjusting the difficulty parameter in the code, students observe a direct correlation with the Block Time (the interval between blocks), validating the theoretical relationship between target difficulty and network throughput.
\end{itemize}

\subsection{Learning Outcome}
This module provides the students with a demonstration of how computational power or capital power (staked assets) can be used to secure a distributed system. Students also learn more about the Byzantine fault tolerance explained in section \ref{ch2_consensus_bft}. And by replacing the consensus engine of the blockchain, they also gain a deeper understanding of the different design patterns used to split the concerns in the codebase and how modular design allows complex systems to evolve without rewriting core logic.

\section{Module 3: Performance Analysis and Benchmarking} \label{ch5_module3}

With the network operational and the consensus mechanism selectable (from Module 2), the curriculum shifts focus to \textbf{System Evaluation}. In distributed systems, theoretical correctness is insufficient; engineers must verify that the system can handle the required throughput and latency constraints. This module introduces students to the methodology of benchmarking blockchain systems, moving beyond simple functional verification to rigorous quantitative analysis.

\subsection{Problem Statement}
Blockchain systems are inherently constrained by the "Blockchain Trilemma" (Scalability, Security, Decentralization). To optimize a system, engineers must first be able to measure it. The core problems addressed in this module are:
\begin{enumerate}
    \item \textbf{Throughput Visibility:} How do we measure Transactions Per Second (TPS) in a distributed asynchronous system where no single global clock exists?
    \item \textbf{Latency Analysis:} How long does it take for a transaction to propagate from the client, through the Mempool, to a finalized block?
\end{enumerate}

\subsection{Learning Objectives}
By completing this module, students will be able to:
\begin{itemize}
    \item Implement \textbf{Instrumentation Logic} within the Python codebase to capture precise timestamps for block creation and transaction finality.
    \item Develop a \textbf{Stress Test Script} to flood the network with high-volume traffic, simulating a "DDoS" or high-usage scenario.
    \item Analyze and visualize performance metrics to empirically quantify the performance trade-off between Proof of Stake (PoS) and Proof of Work (PoW) \cite{croman2016scaling}.
\end{itemize}

\subsection{Tasks}

\textbf{Task 1: System Instrumentation} \\
Students are provided with the \texttt{blockchain/utils/logger.py} and \texttt{node.py} files. They must inject timing logic to capture two critical metrics:
\begin{enumerate}
    \item \textbf{Block Propagation Time:} The delta between a block's timestamp and when it is received by a peer.
    \item \textbf{Transaction Throughput:} The number of transactions processed divided by the block time.
\end{enumerate}
Students must modify the \texttt{handle\_block} method to log these statistics to a structured JSON file for later analysis.

\textbf{Task 2: The Stress Test} \\
Students must write a Python script (extending the provided \texttt{sample\_transactions.py}) that generates hundreds to thousands of valid transactions and broadcasts them to the REST API as rapidly as possible. This "Transaction Spam" forces the node to fill its Mempool, allowing students to observe how the network handles congestion.

This module empirically demonstrates the difference between \textbf{Protocol Limits} (Block Size/Time) and \textbf{Hardware Limits} (CPU/Bandwidth). \section{Module 4: Application Layer - The Library Management System} \label{ch5_module4}

Up to this point, the blockchain has served as a generic ledger for transferring currency.  In this module, students transform the system into the Library Management System described in the thesis proposal. This module introduces the concept of Asset Tokenization and application-specific blockchain systems.

\subsection{Problem Statement}
A standard cryptocurrency transaction (`Transfer 5 Coins`) is insufficient for managing physical inventory like library books. The core problems are:
\begin{enumerate}
    \item \textbf{State Representation:} How do we represent a specific book (e.g., "Introduction to Algorithms") on the blockchain?
    \item \textbf{Logic Constraints:} How do we ensure a book cannot be borrowed if it is currently checked out (We will call this problem the Double-Borrow)?
\end{enumerate}

\subsection{Learning Objectives}
By completing this module, students will be able to:
\begin{itemize}
    \item Extend the core \texttt{Transaction} class to support arbitrary data payloads (Smart Contract-like behavior).
    \item Implement \textbf{Business Logic Validation} in the node to reject invalid state transitions (e.g., returning a book that was not borrowed).
    \item Demonstrate the utility of blockchain for \textbf{Provenance Tracking} by auditing the borrow history of a specific asset \cite{swan2015blockchain}.
\end{itemize}

\subsection{Tasks}

\textbf{Task 1: Transaction Payload Extension} \\
Students must modify the \texttt{Transaction} class in \texttt{blockchain/transaction/transaction.py}. They need to add a \texttt{model} field to the transaction structure that supports types beyond `TRANSFER`, specifically `BORROW` and `RETURN`, and includes a `book\_id`.

\textbf{Task 2: Validation Logic} \\
Students must modify the `AccountModel` to track not just balances, but the "Inventory State." They must implement a check: \textit{If type is BORROW, ensure the book\_id is currently held by the Library Account.}
