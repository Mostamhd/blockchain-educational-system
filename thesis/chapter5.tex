\chapter{Educational Curriculum Design} \label{ch5_curriculum}
\markboth{Educational Curriculum Design}{Educational Curriculum Design}


Having established the technical implementation of the modular blockchain system in Chapter \ref{ch4_implementation}, this chapter presents a comprehensive \textbf{5-Module Laboratory Curriculum}. The core contribution of this thesis is not only the blockchain software developed, but the educational curriculum that utilizes this software to bridge the gap between theoretical distributed systems concepts and practical engineering skills.


The curriculum is designed to transform the learning process from understanding the theoretical aspects to applying what they have learned in their courses or projects by actively engaging with the underlying mechanics of consensus, network propagation, and state management. By interacting with the blockchain system developed in the previous chapter, students can transition from passive knowledge consumers to active system architects.


\section{Approach and Methodology} \label{ch5_pedagogy}

The design of the laboratory exercises is inspired by the theory of Constructionist Learning, proposed by Seymour Papert. This theory states that learning is most effective when students are actively engaged in constructing tangible objects in the real world, rather than acquiring knowledge through passive transmission or consumption \cite{papert1991situating}. In the context of Computer Science education, this necessitates moving beyond theory, static simulations, or black-box interactions with production networks (such as sending transactions on a public Testnet) toward deep system understanding, enabling students to fully understand the concepts by practice and gain experience with code-level modifications.

To facilitate this, the curriculum adopts a Problem-Based Learning methodology. Rather than editing a finished system, students are presented with engineering challenges that mirror real-world distributed systems problems. To ensure a scientific approach, each of the five modules follows a standardized structure inspired by the Active Learning framework \cite{freeman2014active}:


\begin{enumerate}
    \item \textbf{Problem Statement:} Introduces a specific theoretical challenge (e.g., "How do decentralized nodes agree on time?") or an operational deficiency (e.g., "How does the network behave when a peer disconnects?").
    \item \textbf{Learning Objectives:} Defines specific technical competencies the student will acquire by completing the module, aligned with the implementation details discussed in Chapter \ref{ch4_implementation}.
    \item \textbf{Tasks:} This is the core phase, where students are tasked to find a solution and modify the codebase, consensus parameters, altering the network module, or implementing new transaction types to resolve the stated problem.
    \item \textbf{Analysis of Findings:} Record an observation of the system's behaviors in relation to the actions taken from the previous steps, such as throughput, latency, or change in the state of the system.
    \item \textbf{Learning Outcome:} In this phase, students correlate and connect their experimental results with the theoretical bounds established in Chapter \ref{ch2_theoretical_framework}, solidifying their gained knowledge of distributed systems and blockchains.
\end{enumerate}

This structured approach addresses the educational disconnect identified in Chapter \ref{ch1_introduction}, providing a safe environment where breaking and modifying the system facilitates constructive learning rather than causing system failure, thus fostering deeper engagement of both students and teachers in the learning process.

\section{Repository Structure and Setup} \label{ch5_repo_structure}

To support the modular nature of the curriculum, a dedicated GitHub repository has been organized to provide distinct entry points for each learning module. \url{https://github.com/Mostamhd/blockchain-educational-system} \label{ch5_repo_url}

The repository is structured to provide the necessary code base to start working on each module for the students. For each module, two directories are provided:
\begin{itemize}
    \item \textbf{Starter/}: This directory contains the baseline code required to initiate the module. Although the provided code is functional, it is intentionally incomplete. It serves as a starting point from which students are expected to implement the required features and address the problem statement associated with the module.

    \item \textbf{Solution/}: This directory contains a reference implementation corresponding to each module. It is intended to support self-assessment and solution validation. Due to space limitations, the complete solution implementations are not reproduced in full within this thesis. Instead, only the most critical modifications and design decisions are discussed, with the full reference implementations available in the accompanying repository.
\end{itemize}

\textbf{Note:} While the full solutions are provided to ensure no student remains blocked, they are strongly encouraged to attempt each task independently before consulting the \texttt{Solution/} directory. The primary educational value of the curriculum lies in the process of system design, implementation, and debugging, rather than in the examination of completed code alone.

\section{Module 1: Network Fundamentals} \label{ch5_module1}

This module focuses on two foundational building blocks of blockchain technology: creating a custom blockchain via forking and understanding Peer-to-Peer (P2P) discovery. Students are provided with the \textit{Starter Code} for the node software, which contains the core blockchain logic but lacks the specific configuration and implementation required to solve the tasks. The student acts as a system architect, establishing a fork of the blockchain network, and understanding how the network's peer discovery layer works to create a new node and link it to the existing blockchain network.

\subsection{Problem Statement}
To facilitate a comprehensive understanding of blockchain networks, two fundamental challenges must be addressed:

\begin{enumerate}
    \item \textbf{Blockchain Forking:} How can a distinct blockchain network be instantiated such that it is cryptographically incompatible with other networks operating the same software version?
    
    \item \textbf{Peer Discovery Mechanism:} How do initially isolated nodes discover other participants within the network and establish peer-to-peer communication channels?
\end{enumerate}

\subsection{Learning Objectives}
By completing this module, students will be able to:
\begin{itemize}
    \item Execute a \textbf{Hard Fork} by modifying the Genesis configuration, thereby ensuring the cryptographic isolation of their private network.
    \item Understand the mechanics of \textbf{Peer-to-Peer Discovery} and how nodes bootstrap connections to an existing blockchain.
    \item Utilize \textbf{Docker} and \textbf{Docker Compose} to manage containerized nodes, gaining practical experience with containerization and node discovery in P2P systems.
\end{itemize}

\newpage

\subsection{Tasks}

\subsubsection*{Task 1: The Hard Fork}
To establish a unique blockchain network identity, perform the following steps:
\begin{enumerate}
    \item Open the file \texttt{blockchain/blockchain/block.py}.
    \item Locate the method \texttt{genesis()}.
    \item Modify the second parameter of the \texttt{Block} constructor (the \texttt{last\_hash} field) from the default value to a unique identifier (e.g., \texttt{"genesis\_group\_05"} or your Student ID).
    \item Save the file. This change ensures that the hash of Block 0 is mathematically distinct from the original codebase, effectively creating a hard fork.
\end{enumerate}

\subsubsection*{Task 2: Network Node Addition} 
Extend the existing network nodes by adding a new node, \texttt{node\_new}. You may choose to run the node manually via the terminal or orchestrate it via Docker.
\begin{enumerate}
    \item \textbf{Configuration:} Open the \texttt{docker-compose.yml} file.
    \item \textbf{Service Definition:} Define a new service entry named \texttt{node\_new}.
    \item \textbf{Port Mapping:} Map a unique host port to the container's internal P2P port.
    \item \textbf{Discovery:} Update the environment variables to ensure the new node is discoverable.
    \item \textbf{Execution:} Run \texttt{docker-compose up} and verify in the logs that \texttt{node\_new} successfully establishes a handshake with the existing peers.
\end{enumerate}

The complete reference implementation for this module is archived in the accompanying repository, specifically within the \texttt{Module 1} directory (refer to Section \ref{ch5_repo_structure} for repository structure).\newline

\subsubsection*{Task 1: The Hard Fork}
By changing the second parameter in the Genesis block to genesis\_new\_student this change in terms serves as the anchor for the entire blockchain. In the validation logic (\texttt{blockchain.py}), nodes verify that a new block links cryptographically to the previous one. By changing the genesis data, the hash of the block changes, subsequently invalidating all following blocks. Consequently, any node running the original code will calculate a different hash chain, which in turn successfully creates a new isolated blockchain network with a completely different chain of blocks.

\begin{lstlisting}[language=Python, caption=Genesis Block Modification, label=lst:mod1_genesis]
@staticmethod
def genesis():
    return Block([], "genesis_new_student", "genesis", 0)
\end{lstlisting}

\subsubsection*{Task 2: Network Node Addition}
Two approaches may be used to add a node to the system. The first approach consists of executing the \texttt{run\_node.py} script with the appropriate parameters. However, this method requires that all nodes on the same machine be executed in separate terminal sessions. Although functional, this approach can be more difficult to manage as the number of nodes grows.

The second approach, which is adopted in this work, relies on a \texttt{docker-compose} configuration to orchestrate multiple isolated nodes. Each node is represented as a Docker service. Implementing this approach requires the definition of specific environment variables to allow the peer-to-peer (P2P) module to correctly resolve and map Docker container IP addresses and ports to the corresponding network interfaces on the machine. The following configuration example demonstrates the correct method for creating a node in the system using docker-compose.

\begin{lstlisting}[caption=Docker Compose code snippet, label=lst:mod1_docker]
node_new:
    build: ./blockchain
    command: python run_node.py --ip=0.0.0.0 --node_port=8015 
             --api_port=8055 --key_file=./keys/node_new_key.txt
    entrypoint: ['/usr/src/node/entrypoint.sh']
    volumes:
      - ./blockchain:/usr/src/node
    environment:
      - USE_DOCKER=True
      - DOCKER_NODE_CONTAINER_NAMES=node_one,node_two,node_new
      - DOCKER_NODE_PORTS=8010,8011,8015
    labels:
      com.datadoghq.ad.logs: '[{"service": "blockchain_node", 
                                "source": "python"}]'
    ports:
      - 8015:8015
      - 8055:8055
\end{lstlisting}

\textbf{Technical Justification of Configuration Parameters:}
\begin{itemize}
    \item \texttt{command}: Defines the runtime arguments. \texttt{--ip=0.0.0.0} binds the node to all network interfaces within the container, which is strictly necessary for it to receive external TCP connections from other Docker containers.
    \item \texttt{environment}: The environment variables are critical for the \textbf{Peer Discovery} mechanism. In \texttt{p2p\_communication.py}, the \texttt{SocketConnector} class parses these lists to map container names (e.g., `node\_one`) to their respective ports. This allows the new node to identify other nodes in the network and initiate the first handshake.
    \item \texttt{ports}: Maps the internal container ports to the host machine.
\end{itemize}

\subsection{Analysis of Findings}
Upon the completion of the task, students perform the following verifications:

\begin{itemize}
    \item \textbf{Fork verification:} Students compare the "Block Height" and "Last Hash" in the API of their new node versus a node from the original unmodified codebase. They observe that while both nodes are running, both nodes run on different blockchains. This confirms that the modification to the Genesis configuration isolated the network, preventing the "Foreign" chain from corrupting the local ledger.
    \item \textbf{Connectivity Verification:} Students execute \texttt{docker-compose up} and inspect the logs. A successful node creation is indicated when \texttt{node\_new} logs a "Handshake" message, confirming it has established a connection with the other nodes in the blockchain network.
\end{itemize}

\subsection{Learning Outcome}
This module attempts to help students with demonstrating the importance of hashing in blockchains. Students are guided to learn that a blockchain's identity is not defined by the software running it, but rather by the shared history (Blocks) agreed upon by the network peers. Furthermore, the exercise provides practical experience in Infrastructure setup and network topology, by showing how creating a node requires an understanding of the underlying P2P networking layer and node discovery mechanisms in the distributed blockchain systems.

\section{Module 2: Consensus Mechanisms} \label{ch5_module2}

Having established a working distributed network in Module 1, the focus shifts to the mechanism of consensus in decentralized systems. As explained in Section \ref{ch2_consensus}, the \textit{Starter Code} for this module includes the fully containerized setup from \ref{ch4_implementation}, which operates on a simplified POS consensus model. Students are tasked to redesign the system's consensus algorithm to help with achieving consensus in distributed systems, secure the network against attacks, and solve the Byzantine fault tolerance problem. While students are encouraged to explore various algorithms (e.g., Delegated Proof of Stake, Proof of Elapsed Time), this guide focuses on implementing Proof of Work (PoW) as it's still the most used and battle-tested consensus algorithm to date, and due to its educational value in demonstrating the trade-off between computational intensity and scalability.

\subsection{Problem Statement}
In an open, permissionless network where nodes can join anonymously, two critical vulnerabilities must be addressed:
\begin{enumerate}
    \item \textbf{The Sybil Attack:} Without a cost to participate, a single malicious actor can spawn thousands of nodes to gain a majority influence over the network, as mentioned in section \ref{ch2_security_risks}.
    \item \textbf{Leader Selection:} How can the network select a unique leader to propose the next block in a deterministic way without a central coordinator?
\end{enumerate}

\subsection{Learning Objectives}

Upon completion of this module, students will be able to:

\begin{itemize}
    \item Demonstrate an understanding of the architecture of the \texttt{Blockchain} class by designing a mechanism to switch the consensus algorithm of the prototype system implemented in Chapter~\ref{ch4_implementation} to the Proof of Work (PoW) consensus model described in Section~\ref{ch2_consensus_pow}.
    \item Design and implement a cryptographic \textbf{Proof of Work} algorithm by translating theoretical concepts—such as nonce, difficulty, and target—into executable Python code.
    % \item Measure and analyze the impact of consensus choices on system performance.
\end{itemize}

\subsection{Tasks}

\subsubsection*{Task 1: The Consensus Implementation}
To secure the network, you must change the consensus algorithm and implement a Proof of Work algorithm to achieve consensus in the blockchain network.
\begin{enumerate}
    \item Create a new class \texttt{ProofOfWork} (e.g., in \texttt{blockchain/blockchain/pow/proof\_of\_work.py}) that adheres to the system's consensus interface.
    \item Implement the \texttt{mine(block)} method to perform the "Mining" process described in Section \ref{ch2_consensus_pow}. This method must:
    \begin{itemize}
        \item Initialize the block's \texttt{nonce} to 0.
        \item Execute a \texttt{while} loop that calculates the cryptographic hash of the block.
        \item Check if the resulting hash meets the difficulty target (e.g., starts with four leading zeros: \texttt{'0000'}).
        \item Increment the \texttt{nonce} and repeat until a valid hash is found.
    \end{itemize}
\end{enumerate}

\subsubsection*{Task 2: The Mining Loop Integration} 
The default node behavior is designed for Proof of Stake (Time-Bound), which waits for specific slots. You must alter the \texttt{node.py} logic to support the continuous, CPU-Bound nature of Proof of Work.
\begin{enumerate}
    \item Open the \texttt{blockchain/blockchain/node.py} file and locate the \texttt{produce\_block} method.
    \item Modify the production loop to check which consensus strategy is currently active.
    \item Implement the conditional logic for PoW:
    \begin{itemize}
        \item Instead of waiting for a turn, the node must continuously call the \texttt{mine()} method created in Task 1.
        \item Ensure the system blocks (waits) until the mining function returns a valid block.
        \item Immediately broadcast the mined block to the network upon success.
    \end{itemize}
\end{enumerate}

\subsubsection*{Task 1: The Consensus Implementation}
The solution involves creating a class that wraps the hashing logic in a while loop.

\begin{lstlisting}[language=Python, caption=Proof of Work Mining Logic code snippet, label=lst:mod2_pow_class]
class ProofOfWork:
    def __init__(self, difficulty=4):
        self.difficulty = difficulty

    def mine(self, block):
        block.nonce = 0
        while True:
            hash = block.hash_block()
            if hash.startswith('0' * self.difficulty):
                return block
            block.nonce += 1
\end{lstlisting}

The \texttt{mine} method models the block creation process as a probabilistic brute-force search problem. The \emph{difficulty} parameter serves as a regulatory control variable: incrementing the difficulty by one results in an exponential increase in the expected number of iterations required to discover a valid solution.

As a consequence, the effective \enquote{cost} of executing the attacks described in Section~\ref{ch2_security_risks} becomes proportional to the attacker’s available physical computing resources and the associated hardware costs, rather than merely the number of IP addresses under the attacker’s control.

\subsubsection*{Task 2: The Mining Loop Integration}
The Node class must be updated to handle the blocking nature of the mining process.

\begin{lstlisting}[language=Python, caption=Integrating the Mining Loop snippet, label=lst:mod2_node_integration]
def run_produce_block(self):
    while self.keep_running:
        if isinstance(self.blockchain.consensus, ProofOfWork):
            block = self.blockchain.create_block(
                self.transaction_pool.transactions, self.wallet
            )
            mined_block = self.blockchain.consensus.mine(block)
            
            if mined_block:
                self.p2p.broadcast(mined_block)
        else:
            time.sleep(10)
\end{lstlisting}

\section{Module 3: Performance Analysis and Benchmarking} \label{ch5_module3}
Once the networking layer is operational and a consensus mechanism has been selected after completing the previous two modules, the curriculum shifts its focus to \textbf{System Evaluation}. In the context of distributed systems and blockchains specifically, theoretical correctness alone is insufficient. Engineers must also verify that the system satisfies the required throughput and latency constraints.

This module introduces students to systematic methodologies for benchmarking blockchain systems. By doing so, it deepens their understanding of the interactions between the system’s components. It encourages a transition from basic functional validation toward an analytical evaluation of the system performance.

\subsection{Problem Statement}
Blockchain systems are inherently constrained by the Blockchain Trilemma (Scalability, Security, Decentralization) \cite{buterintrilemma}. To optimize a system, engineers must first be able to measure its performance. The core problems addressed in this module are:
\begin{enumerate}
    \item \textbf{Throughput:} How do we measure Transactions Per Second (TPS) in a distributed system where no single central service exists?
    \item \textbf{Latency:} How long does it take for a transaction to move and propagate from the client, through the Mempool, to a finalized block?
\end{enumerate}

\subsection{Learning Objectives}
By completing this module, students will be able to:
\begin{itemize}
    \item Implement the logic within the Python codebase to capture the precise timestamps for block creation and transaction finality.
    \item Develop a Stress Test script which floods the blockchain network with high-volume traffic, simulating a DDoS or high-usage scenarios.
    \item Analyze and visualize performance metrics to empirically quantify the performance trade-off between Proof of Stake (PoS) and Proof of Work (PoW) \cite{croman2016scaling}.    
\end{itemize}

def handle_block(self, block):

    tps = len(block.transactions) / 10.0

def stress_test(endpoint, count=500):
    print(f"Starting Stress Test: {count} transactions...")
    start_time = time.time()
    
    for i in range(count):
        tx = wallet.create_transaction(receiver_key, 1, "TRANSFER")

\subsection{Analysis of Findings}

Upon completion of the tasks, students conduct a two-step verification process to validate both the correctness of their implementation and the behavior of the network under load.

\textbf{1. Throughput Verification:}
Before executing the full stress test, students verify that the implemented logging mechanism correctly captures relevant network metrics.

\begin{itemize}
     \item \textbf{Metric Validity:}  
    Students inspect the generated log files after the production of several blocks under normal operating conditions. They verify that the Block Propagation Time is recorded as a correct non-zero latency value and that throughput is computed correctly based on the number of transactions contained within each received block.

    \item \textbf{Data Integrity:}  
    Students confirm that the JSON output is well-formed and syntactically valid, ensuring that it can be reliably parsed and analyzed.

\end{itemize}

\textbf{2. Stress Test Analysis:}
Students execute a high-load simulation under two distinct network configurations: one operating with the default \textbf{Proof of Stake} (PoS) consensus mechanism and another employing the \textbf{Proof of Work} (PoW) implementation developed in Module~2.
\begin{itemize}
    \item \textbf{PoS Performance:}  
    The system maintains a stable block production rate throughout the experiment. However, students can observe that the mempool is cleared gradually due to the fixed block size parameters set for each block. As a result, the observed \textit{transactions per second} (TPS) is constrained primarily by protocol parameters and configurations.

    \item \textbf{PoW Performance:}  
    A degradation in network latency is observed. Because the node’s processing is largely consumed by the mining process, leading to transaction requests timeouts or significantly delayed propagation. This behavior provides a practical demonstration of the computational bottleneck inherited by Proof of Work–based consensus mechanisms.
\end{itemize}

\subsection{Learning Outcome}

This module aims to illustrate the difference between protocol-imposed limits (e.g., block size and block interval) and hardware-imposed limits (e.g., CPU capacity). Students gain an understanding that, while Proof of Stake is computationally efficient, it remains bounded by the network’s ability to disseminate data. In contrast, Proof of Work is fundamentally constrained by the distributed hashing consensus performance, resulting in lower scalability for the system.

\section{Module 4: Application Layer and Smart Assets} \label{ch5_module4}

While previous modules focused on the infrastructure aspects of the system—namely, networking, consensus, and performance, this module shifts the focus to the \textbf{Application Layer}. In this stage, students transition from analyzing the blockchain as a mechanism for currency transfer to utilizing it as a secure and immutable state machine capable of storing and managing arbitrary application data.

Rather than serving exclusively as a financial ledger, the blockchain is introduced as a general-purpose platform for enforcing state transitions under decentralized trust assumptions.

\subsection{Problem Statement}

Cryptocurrencies such as Bitcoin employ a UTXO-based data model that is optimized for currency tracking, as discussed in Section~\ref{ch2_relatedWork_bitcoin}. However, general-purpose applications require more expressive data structures to represent and manage digital assets. The central challenge addressed in this module is therefore defined as follows:

\begin{quote}
\textit{How can arbitrary business logic and structured application data be embedded within blockchain transactions to implement a decentralized Library Management System?}
\end{quote}

\subsection{Learning Objectives}
Upon successful completion of this module, students will be able to:
\begin{itemize}
    \item Extend the core \textbf{transaction schema} to support arbitrary data payloads beyond simple value transfers.
    \item Implement \textbf{business logic validation} to ensure that only semantically valid state transitions (e.g., unique ISBN identifiers) are accepted into the ledger.
    \item Interact with the blockchain to perform application-level state transitions, including the registration and transfer of ownership of digital assets.
\end{itemize}

\newpage

\subsection{Tasks}

\subsubsection*{Task 1: Hello World Transaction}
Before building complex assets, students first enable the blockchain to store general textual data.

\begin{enumerate}
    \item Open \texttt{blockchain/blockchain/transaction/transaction.py}.
    \item Modify the \texttt{\_\_init\_\_} method of the \texttt{Transaction} class to accept an optional \texttt{data} parameter, defaulting to \texttt{None}.
    \item Update the \texttt{to\_dict()} serialization method to include the new \texttt{data} field in the resulting JSON representation.
    \item Validate the implementation by submitting a transaction via the API with the payload \texttt{"Hello Blockchain"} and verifying it appears in the next propagated block.
\end{enumerate}

\subsubsection*{Task 2: Asset Schema}
Students subsequently define a structured digital asset representing a book.

\begin{enumerate}
    \item Create a new class \texttt{Book} in the file \texttt{digital\_asset.py}.
    \item Define the schema attributes: \texttt{ISBN} (unique identifier), \texttt{Title}, \texttt{Author}, and \texttt{Owner\_Public\_Key}.
    \item Implement a \texttt{to\_json()} method that serializes the book object into a string format suitable for inclusion in the transaction \texttt{data} field introduced in Task~1.
\end{enumerate}

\paragraph{Task 3: Library Management System}
The final task consists of implementing the application-level business logic.

\begin{enumerate}
    \item \textbf{Registration:}  
    Create a script named \texttt{register\_book.py} that instantiates a \texttt{Book} object, signs a transaction containing its book's data, and broadcasts the transaction to the network.

    \item \textbf{Transfer:}  
    Create a script named \texttt{transfer\_book.py} that generates a transaction updating the \texttt{Owner\_Public\_Key} of an existing book, identified by its ISBN.

    \item \textbf{Validation:}  
    Execute the scripts and confirm, via the Blockchain data, that the books are correctly registered and transferred on the ledger.
\end{enumerate}

The complete reference implementation for this module is archived in the accompanying repository, specifically within the \texttt{Module 4} directory (refer to Section \ref{ch5_repo_structure} for repository structure).\newline

\subsubsection*{1. Task 1: Transaction Schema Modification} 
To support application-specific transactions, the transaction class must include a generic data field, which allows the transaction structure to become flexible enough to support any application logic without changing the underlying consensus mechanism.

class Transaction:

                        signature=None, data=None):
        self.sender = sender
        self.receiver = receiver
        self.amount = amount
        self.data = data 
        self.signature = signature
\end{lstlisting}

\subsubsection*{2. Task 2: The Digital Asset Class}

\textbf{C. Verification:}
Since the blockchain is an append-only log, the system cannot simply read the current owner from a database row. Instead, it must validate the current state and action before updating the data .

\begin{lstlisting}[language=Python, caption=State Reconstruction Logic, label=lst:mod4_verify]
def get_current_owner(blockchain_history, target_isbn):
    current_owner = None
    
    for block in blockchain_history:
        for tx in block.transactions:
            if not tx.data: continue 
            try:
                payload = json.loads(tx.data)
            except: continue
            
            if payload.get("isbn") == target_isbn:
                action = payload.get("action")
                
                if action == "REGISTER":
                    if current_owner is None:
                        current_owner = payload.get("owner")
                        
                elif action == "TRANSFER":
                    if tx.sender == current_owner:
                        current_owner = payload.get("new_owner")
                        
    return current_owner
\end{lstlisting}

\begin{itemize}
    \item \textbf{Data Persistence:}  
    By querying the API endpoint \texttt{/get\_block/\{last\_index\}}, students observe that the JSON-encoded book data is permanently recorded within the block payload.

    \item \textbf{Immutability:}  
    Students observe that if a node attempted to modify a book’s title within a locally altered block, students observe that hash validation will fail, as the data of the block is used in generating the hash of the block, which reinforces the principle that once a library record is committed to the ledger, it cannot be tampered with.
\end{itemize}

By building the Library Management System, they gain the foundational skills for understanding decentralized applications (dApps) and learn how complex application schemas can be mapped onto transaction structures.

\section{Module 5: System Extensibility and Design Challenge} \label{ch5_module5}

This final module moves beyond system implementation and introduces students to \textbf{architectural criticism and extensibility analysis}. Having developed a comprehensive understanding of the blockchain system across the previous modules, students are now required to critically evaluate its limitations and propose scalable design alternatives suitable for real-world deployment. Consequently, this module also functions as an open-ended future work extension of the implemented system.

\subsection{Problem Statement}
In the previous module, students stored book metadata (e.g., title and author) directly on the blockchain. While this approach is acceptable for small, structured data, it exposes a fundamental scalability limitation when applied to large data. Specifically, the question arises:
\textit{What are the implications of storing the complete PDF of a book directly on the blockchain?}

Blockchains replicate data across all participating nodes. As a result, storing large files on the order of megabytes or gigabytes incurs significant storage increase, bandwidth, and synchronization costs. Over time, this leads to unsustainable ledger growth and threatens the ability of nodes to participate in consensus.


\subsection{Learning Objectives}
Upon successful completion of this module, students will be able to:
\begin{itemize}
    \item Analyze the \textbf{storage and bandwidth costs} associated with distributed ledgers.
    \item Design a \textbf{hybrid on-chain/off-chain architecture} that preserves integrity while improving scalability.
    \item Explain the role of decentralized storage systems such as the \textbf{InterPlanetary File System (IPFS)} in blockchain-based applications.
\end{itemize}

\newpage
\subsection{Tasks}

\subsubsection*{Task 1: Protocol Defense (Block Size Limit)}
To prevent ledger bloat and ensure propagation latency remains within acceptable bounds, the protocol must strictly enforce resource limits.
\begin{enumerate}
    \item Open \texttt{blockchain/blockchain/blockchain.py}.
    \item Define a constant \texttt{MAX\_BLOCK\_SIZE\_BYTES = 2048} (2KB) to simulate a constrained environment.
    \item Locate the \texttt{add\_block()} or \texttt{validate\_block()} method.
    \item Implement logic to serialize the candidate block, measure its size in bytes, and raise a \texttt{BlockSizeExceededError} if it violates the limit.
    \item \textbf{Verification:} Attempt to broadcast a transaction containing a large string payload (e.g., 5KB of text) and verify that the node rejects the resulting block.
\end{enumerate}

\subsubsection*{Task 2: Schema Enforcement (Hash Pointers)}
To support the Hybrid Architecture efficiently, the protocol must enforce that "Storage" transactions contain valid pointers (hashes) rather than raw data.
\begin{enumerate}
    \item Open the transaction validation logic in \texttt{blockchain/blockchain/node.py}.
    \item Implement a validation rule for transactions with the type \texttt{STORE\_FILE}.
    \item The rule must assert that the \texttt{data} payload is a valid SHA-256 hash (i.e., a 64-character hexadecimal string) and not an arbitrary file blob.
    \item \textbf{Verification:} Submit a transaction with type \texttt{STORE\_FILE} containing a raw file string. Observe the validation failure in the logs.
\end{enumerate}

\newpage
\subsection{Reference Solution and Analysis}

\subsubsection*{Task 1: Enforcing Block Size Limits} 
This solution demonstrates how to protect the network layer at the consensus level. By serializing the block before acceptance, the node ensures no peer can burden the network with bandwidth-heavy data.

\begin{lstlisting}[language=Python, caption=Block Size Enforcement, label=lst:mod5_blocksize]

MAX_BLOCK_SIZE = 2048

def add_block(self, block):
    block_bytes = json.dumps(block.to_dict()).encode('utf-8')
    
    if len(block_bytes) > MAX_BLOCK_SIZE:
        logger.error(f"Block rejected: Size {len(block_bytes)}B "
                     f"exceeds limit {MAX_BLOCK_SIZE}B")
        return False
        
\end{lstlisting}

\subsubsection*{Task 2: Schema Validation for Off-Chain Pointers}
This implementation forces the application layer to utilize off-chain storage. The protocol explicitly forbids "Data Dumping" by validating the format of the payload.

\begin{lstlisting}[language=Python, caption=Transaction Schema Validation, label=lst:mod5_schema]

def validate_transaction(self, tx):
    if tx.type == "STORE_FILE":
        if not re.match(r'^[a-fA-F0-9]{64}$', tx.data):
            raise InvalidTransactionError(
                "invalid hash"
            )
    
\end{lstlisting}


\subsection{Analysis of Findings}
By implementing these constraints, students observe the transition from a "Passive" ledger to an "Active" protocol:
\begin{itemize}
    \item \textbf{Rejection of Bloat:} Students confirm that while they \textit{can} technically create a 5MB transaction in the client, the network nodes autonomously reject it, preserving the health of the P2P mesh.
    \item \textbf{Standardization:} The system now enforces a data standard. The blockchain acts as the "Notary" verifying the \textit{existence} of data (via the hash) without bearing the cost of \textit{storing} it.
\end{itemize}

\subsection{Learning Outcome}
This module aims to show students that blockchain technology is not a universal solution for data storage but a technology within a broader system architecture. By critically evaluating design trade-offs, students are introduced to advanced scalability concepts. This prepares students for advanced topics such as Layer 2 scaling and decentralized storage \cite{poon2017plasma, buterin2021rollups, benet2014ipfs}.
